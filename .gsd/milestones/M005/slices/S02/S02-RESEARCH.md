# S02: Page scaffold, sidebar, and source map — Research

**Date:** 2026-03-19

## Summary

This is straightforward scaffolding work following established patterns. The codebase already has a complete model for everything S02 needs: `manage-pages.mjs` shows how to scaffold MDX stubs, manipulate `astro.config.mjs` sidebar entries, and update `page-source-map.json`. `build-page-map.mjs` shows how source mappings are built programmatically. The `prompts.json` (produced by S01) provides the exact slug list, group taxonomy, and source filenames needed.

The three deliverables — 32 stub MDX files, sidebar with 4 sub-groups, page-source-map with 32 entries — are independent write operations that can be built and verified in sequence. No new libraries, no unfamiliar APIs, no ambiguous requirements.

The one structural difference from commands is that Prompts needs **nested sub-groups** in the sidebar (4 groups within a "Prompts" top-level section). Starlight 0.38 supports nested `{ label, items }` — the current sidebar only uses one level of nesting but the framework handles arbitrary depth.

## Recommendation

Build in 3 tasks following the established patterns exactly:

1. **Stub MDX pages** — generate 32 files in `src/content/docs/prompts/` using the same scaffold pattern as `createNewPages()` in `manage-pages.mjs`. Read `prompts.json` for the slug list. Each stub gets frontmatter with title and description, plus the `:::caution` scaffold notice.

2. **Sidebar config** — add a "Prompts" section to `astro.config.mjs` with 4 nested sub-groups (Auto-mode Pipeline, Guided Variants, Commands, Foundation), each containing the slugs from `prompts.json` in the corresponding group. Insert the Prompts section after the Commands section.

3. **Page source map + build-page-map** — add a section to `build-page-map.mjs` that generates entries for all 32 prompt pages. Each prompt page's sole dependency is `src/resources/extensions/gsd/prompts/{name}.md`. Then regenerate `page-source-map.json`.

Verify with `npm run build` (should succeed with 32 new pages) and test suite updates.

### Existing Patterns to Follow

**Stub MDX pattern** (from `createNewPages()` in `manage-pages.mjs`, line ~230):
```
---
title: "{label}"
description: "Documentation for {label}"
---

:::caution
This page is a scaffold — content has not been generated yet.
:::
```

For prompts, the title should be the prompt name (e.g. "execute-task"), and the description should reference its role (e.g. "Prompt reference: execute-task").

**Sidebar nested sub-group pattern** (Starlight supports this):
```js
{
  label: 'Prompts',
  items: [
    {
      label: 'Auto-mode Pipeline',
      items: [
        { label: 'execute-task', link: '/prompts/execute-task/' },
        // ...
      ],
    },
    // ... more sub-groups
  ],
},
```

**Page source map entry pattern** (from `build-page-map.mjs`):
```js
// ─── 6. Prompt pages (32) ─────────────────────────────────────────
const promptsJson = JSON.parse(
  fs.readFileSync(path.join(ROOT, "content/generated/prompts.json"), "utf-8")
);
for (const prompt of promptsJson) {
  addPage(`prompts/${prompt.slug}.mdx`, [
    `src/resources/extensions/gsd/prompts/${prompt.name}.md`,
  ]);
}
```

**page-source-map key format** (D058): `prompts/{slug}.mdx` — consistent with `commands/{slug}.mdx`.

**Source dep path format**: `src/resources/extensions/gsd/prompts/{name}.md` — repo-relative, matching manifest.json key format.

## Constraints

- **Starlight internal link format** — all links in prompt stubs (and later real content) must use `../` prefix format (KNOWLEDGE.md rule). Not relevant for stubs but the frontmatter should be correct from the start.
- **Page-source-map key format** — must be `prompts/{slug}.mdx` per D058.
- **Source dep paths** — must be `src/resources/extensions/gsd/prompts/{name}.md` — repo-relative, matching manifest.json keys. The manifest validation in `build-page-map.mjs` (`addPage` helper) will warn if a path isn't in manifest.json.
- **`prompts.json` is the single source of truth** — do not hardcode slug lists or group membership. Read from the generated JSON so changes in S01's extractor automatically propagate.

## Common Pitfalls

- **Hardcoding page count in tests** — the test currently asserts `43` pages but `buildPageSourceMap()` actually produces 48. The test is outdated (5 command pages were added after the test was written: keys, logs, new-milestone, skip, undo). The test will fail before S02 changes are made. The executor must first fix the count to 48, then update to 80 after adding the prompt section. Alternatively, update directly to 80 and fix the COMMAND_SLUGS array (add the 5 missing slugs) in the same edit.
- **Sidebar insertion position** — the Prompts section should go between Commands and Recipes. The closing `},` of the Commands section is around line 69. Insert the new block there. If the line numbers shift due to other edits, anchor on the `label: 'Recipes'` line and insert before it.
- **`build-page-map.mjs` `addPage` validation** — the helper validates source deps exist in `manifest.json`. The prompt `.md` files must be present in the manifest. They should be (they're under `src/resources/extensions/gsd/prompts/` which the manifest tracks), but verify.
