# S02: Astro Site Scaffold with Custom Design

**Running Astro/Starlight site with phosphor-green terminal-native dark theme, custom Header/Footer, Mermaid diagram support, Pagefind search, and a prebuild pipeline that bridges 126 S01 docs into 131 rendered HTML pages.**

## What Happened

Four tasks built the site scaffold bottom-up:

**T01** bootstrapped Astro 6.0.5 + Starlight 0.38.1 with the Mermaid plugin, content collection (docsLoader + docsSchema), and a minimal splash index page. Hit a build-breaking transitive dependency conflict — `@pasqal-io/starlight-client-mermaid` pulls `@astrojs/internal-helpers@0.7.6` but Astro 6 needs 0.8.0. Fixed with an npm override (D018). Site/base configured for GitHub Pages at `gsd-build.github.io/gsd2-guide`.

**T02** applied the terminal-native dark theme (the primary design risk this slice retired). Installed JetBrains Mono Variable (headings/code) and Outfit Variable (body) fonts. Created a two-layer CSS architecture: `custom.css` for Starlight CSS variable overrides (dark-mode: near-black backgrounds, phosphor green #39FF14 accent, green-tinted gray scale; light-mode: inverted green palette) and `terminal.css` for effects (custom scrollbar, scanline overlay, monospace headings with glow, code block enhancements via ExpressiveCode variables, sidebar active-link highlight, table styling). The result is visually distinctive — clearly not default Starlight.

**T03** wired three integration surfaces for downstream slices: (1) Header.astro wrapping Starlight's default Header with a fixed-position version badge (placeholder "v0.0.0" for S05), (2) Footer.astro with "Built with GSD · Powered by Starlight" branding, and (3) `scripts/prebuild.mjs` — the content bridge that reads 126 markdown files from `content/generated/docs/`, extracts `# Heading` lines as YAML frontmatter titles, strips duplicate headings, handles edge cases (no heading, existing frontmatter, YAML-unsafe chars), and writes results to `src/content/docs/`. Uses `.generated-manifest.json` to track its own files for safe cleanup that never touches placeholder pages.

**T04** created placeholder pages demonstrating every component variant: aside types (note/tip/caution/danger), Tabs, CardGrid, LinkCard, Badge variants, Steps, three Mermaid diagram types (flowchart, sequence, state), and ExpressiveCode features (multi-language syntax, line highlighting, file labels, diff markers, terminal frames). Enhanced the index page to a proper hero landing page. Configured sidebar with a Placeholder group plus 7 autogenerate entries for S01 content subdirectories.

## Forward Intelligence

### What the next slice should know
- The site scaffold is fully functional with 131 pages rendering. `npm run dev` and `npm run build` both work. S01 content is already flowing through the prebuild pipeline into the built site.
- Sidebar configuration in `astro.config.mjs` uses `autogenerate` for S01 content groups. S03/S04 will likely need to restructure the sidebar as they add purpose-built pages that replace or supplement the raw prebuild output.
- The Header version badge is at `src/components/Header.astro` — S05 needs to read `content/generated/releases.json` and inject the real version. The badge is a fixed-position element in the top-right, styled with terminal theme variables.
- Placeholder pages in `src/content/docs/placeholder/` show how to use every Starlight component. Reference them when building real pages.

### What's fragile
- **npm override for @astrojs/internal-helpers** — If Astro or starlight-client-mermaid update, this override may need to be removed or updated. Watch for build errors mentioning `collapseDuplicateLeadingSlashes`.
- **Prebuild manifest cleanup** — If a file in `content/generated/docs/` is renamed (not just modified), the old file stays in `src/content/docs/` until the next clean prebuild run. The manifest tracks by path, so renames appear as a delete + create.
- **Scanline overlay** — The `body::before` pseudo-element creates a fixed repeating-gradient overlay. If it causes rendering performance issues on large pages, remove it from `terminal.css` without affecting the theme.

### Authoritative diagnostics
- `npm run build` exit code + page count in stdout — the single most reliable health signal. Should report 131+ pages and exit 0.
- `src/content/docs/.generated-manifest.json` — lists every file the prebuild created. If a page is missing from the build, check this manifest first.
- Browser DevTools computed styles on `html[data-theme='dark']` → `--sl-color-bg` should be `#0a0e0a` and `--sl-color-text-accent` should be `#39ff14`. If these are Starlight defaults, the customCss wiring is broken.

### What assumptions changed
- **Starlight customization ceiling** — Original risk assumption was that terminal-native design might push past Starlight's CSS variable system. In practice, CSS variables + component overrides handled everything needed. The customization ceiling is NOT a blocker for the design ambition.
- **Base path and dist structure** — `base: '/gsd2-guide'` does NOT nest files under `dist/gsd2-guide/`. Files live at `dist/placeholder/components/index.html`, not `dist/gsd2-guide/placeholder/...`. The base path is applied at serve time. Verification commands should check `dist/` paths directly.
- **npm prebuild lifecycle hook** — The plan assumed explicit `&&` chaining for both dev and build. In practice, npm's `prebuild` hook handles the build case automatically, so the `build` script is just `astro build` to avoid running prebuild twice.
