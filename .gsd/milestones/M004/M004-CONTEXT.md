# M004: Claude Code–Powered Documentation Regeneration

**Gathered:** 2026-03-18
**Status:** Ready for planning

## Project Description

A living documentation site for gsd-pi that stays current with the project's extreme release velocity. The site's content was originally generated by Claude Code — reading source files, cross-referencing, making editorial judgments with full tool access. M003 attempted to automate updates using the bare Anthropic API (`@anthropic-ai/sdk`), which has never successfully regenerated a page in production and is fundamentally less capable than what created the original content.

This milestone replaces the API-based approach with `claude -p` (Claude Code's non-interactive mode), which gives the same full tool access that wrote the original docs. When gsd-pi changes, the documentation regenerates at the same quality.

## Why This Milestone

M003 built solid detection infrastructure (page-source-map, diff detection, freshness tracking) but the regeneration layer is wrong. The `@anthropic-ai/sdk` approach sends source files in a prompt and gets back text — no ability to read additional files, check links, verify output, or iterate. The user explicitly wants Claude Code (not API keys) to do the regeneration, matching the original creation process.

The user's core requirement: every page in gsd-guide is traceable to source in gsd-pi, and when source changes, Claude Code regenerates the target page at the same quality it was originally written.

## User-Visible Outcome

### When this milestone is complete, the user can:

- Say "update gsd-guide" and have the pipeline detect stale pages, regenerate them via Claude Code, build, link-check, commit, push to main, and deploy to GitHub Pages — zero intervention

- Environment: local dev (macOS), Claude Code CLI installed and authenticated
- Live dependencies: Claude Code CLI (`claude -p`), npm registry (gsd-pi), GitHub Pages

## Risks and Unknowns

- **claude -p output quality** — Will `claude -p` produce the same quality as interactive Claude Code for documentation pages? It has full tool access but the prompt must be precise enough to direct it. The quality bar is the existing M002 pages.
- **reference/skills.mdx and reference/extensions.mdx have 295 and 289 deps** — These pages map to enormous numbers of source files. A focused `claude -p` invocation with all those files in context may be token-intensive or produce unfocused output. May need a different strategy for reference pages vs command pages.
- **claude -p in a spawned subprocess** — `execSync`/`spawnSync` with the claude binary needs the right env, PATH, and working directory. Timeout behavior needs consideration for long regenerations.
- **Page-source-map accuracy** — All 43 mappings have zero broken deps (every mapped source path exists in manifest.json), but the mappings may still be semantically wrong — mapping to files that don't actually influence the page content.

- `scripts/update.mjs` — 9-step pipeline. The `regenerate` step calls `runRegenerateStale()` which calls `getStalePages()` from `check-page-freshness.mjs`. This wiring stays; only the inner `regeneratePage()` implementation changes.
- `scripts/check-page-freshness.mjs` — Exports `getStalePages()` (compares page-versions.json against manifest.json) and `stampPages()`. Solid — keep as-is.
- `content/generated/page-source-map.json` — 43 pages mapped, 0 broken deps as of 2026-03-18.
- `content/generated/page-versions.json` (committed as `page-versions.json` at root) — Freshness tracking. 43 pages stamped.
- `src/content/docs/commands/capture.mdx` — Exemplar page used in the current system prompt. High-quality reference.
- `package.json` — Has `@anthropic-ai/sdk` as a dependency. Remove it once regeneration is migrated.

## Technical Constraints

- `claude` CLI must be in PATH (it's at `/Users/davidspence/.local/bin/claude`)
- `claude -p` runs with `bypassPermissions` mode by default — no interactive prompts
- Working directory for `claude -p` must be the project root so Claude Code can access all files
- The spawn must not inherit the parent's stdin (avoids TTY issues)
- `reference/skills.mdx` and `reference/extensions.mdx` have 295 and 289 deps — consider whether to pass all deps or use a smarter subset

## Integration Points

- `claude -p` CLI — spawned via `execSync`/`spawnSync` from Node.js; uses the user's existing Claude Code authentication (no API key needed)
- npm registry — `npm install -g gsd-pi@latest` installs the source under the global npm path, which is what the extraction and regeneration scripts read from
- GitHub (`git push origin main`) — already wired into the "update gsd-guide" workflow

## Open Questions

- **Prompt for `claude -p`**: Should the prompt be "rewrite this page from scratch given these source files" or "review this page against current source and update what's changed"? The latter is cheaper and preserves good content, but may miss structural improvements the source now warrants.
- **Reference page deps**: `reference/skills.mdx` has 295 deps (all skill reference files). Should we pass all of them or ask Claude to read them itself via the Read tool?
- **Per-page or batch**: One `claude -p` invocation per page (isolated, clear) vs one session that handles all stale pages (cheaper on context, but one failure affects all).
