# S01: Claude Code Regeneration Engine

**Replaced API-based regeneration with `claude -p` subprocess invocation, removed `@anthropic-ai/sdk` entirely, proved integration on a real page (capture.mdx regenerated by claude-sonnet-4-6 in 140s with correct structure), and established the subprocess mock testing pattern.**

## What Happened

Three tasks rewrote the core regeneration engine from Anthropic SDK to Claude Code subprocess:

**T01 — Core rewrite.** `scripts/lib/regenerate-page.mjs` was fully rewritten. The SDK `client.messages.create()` call was replaced with `spawnSync('claude', ['-p', ...])` using flags `--output-format stream-json`, `--no-session-persistence`, `--dangerously-skip-permissions`, `--model sonnet`. The system prompt carries quality rules (section order, Mermaid styling, frontmatter format, link format) plus the exemplar page (capture.mdx). The user message instructs Claude to read the current page and listed source files, then write updated MDX directly via the Write tool. `findClaude()` detects CLI availability via `execSync('claude --version')` and returns false gracefully when absent. `parseStreamJson()` extracts model name and duration from the subprocess's stream-json output. `capDeps()` substitutes curated compiled JSON paths for reference pages with >50 dependencies (skills→skills.json, extensions→extensions.json, agents→agents.json).

**T02 — Test rewrite.** The old test file mocked `options.client` (SDK injection point that no longer exists). The new 20-test suite uses `tests/fixtures/mock-claude.sh` — a shell script that simulates the claude CLI controlled by environment variables. Tests cover parseStreamJson parsing (6), findClaude detection (3), prompt construction via stdin capture (2), regeneratePage integration (5), and regenerateStalePages batch logic (4).

**T03 — Pipeline wiring and integration proof.** `update.mjs`'s `ANTHROPIC_API_KEY` guard was replaced with `findClaude()`. `@anthropic-ai/sdk` was removed from `package.json` devDependencies. The integration proof ran `node scripts/lib/regenerate-page.mjs commands/capture.mdx` — Claude Code (claude-sonnet-4-6) regenerated the page in 139.7s with valid frontmatter and all 6 required sections. The Astro build passed with 64 pages.

## Forward Intelligence

### What the next slice should know
- `regeneratePage(pagePath, sourceFiles, options)` and `regenerateStalePages(options)` have the same signatures as before. S02 should not need to change any calling code in update.mjs — the wiring is already done.
- The result shape changed: `{ model, durationMs, pagePath }` on success instead of `{ inputTokens, outputTokens, model, elapsedMs }`. The update.mjs logging has already been updated to match.
- `findClaude()` is already wired into update.mjs's `runRegenerateStale()`. S02 just needs to run the pipeline end-to-end.
- Real regeneration takes ~140s per page (capture.mdx took 139.7s with claude-sonnet-4-6). Budget time accordingly for the 3 stale pages.

### What's fragile
- Prompt quality is the biggest risk — the system prompt + user message must produce M002-quality output. capture.mdx worked well, but reference pages with different structures (skills, extensions) may need prompt tuning in S02.
- `--dangerously-skip-permissions` flag is required for non-interactive subprocess invocation. If Claude Code changes this flag name or behavior, the subprocess will fail.
- CWD is set to project root (not worktree) to avoid broken git hooks. If the project moves or the root path calculation changes, subprocess invocation will break.

### Authoritative diagnostics
- `node scripts/lib/regenerate-page.mjs <page>` — single-page debug with ✓/✗/⊘ status, model name, duration
- `node -e "import('./scripts/lib/regenerate-page.mjs').then(m => console.log(m.findClaude()))"` — CLI availability check
- `node --test tests/regenerate-page.test.mjs` — 20-test suite validates all code paths

### What assumptions changed
- Original plan assumed per-page vs batch invocation strategy decision was needed — in practice, per-page invocation via `regenerateStalePages()` calling `regeneratePage()` in sequence is the only strategy. No batch mode was needed.
- D046 (SDK dependency injection via options.client) is superseded by D050 (subprocess invocation). The mock strategy is now a shell script fixture, not a client object.
