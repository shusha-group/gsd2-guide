# M004: Claude Code–Powered Documentation Regeneration

**Replaced API-based documentation regeneration with `claude -p` subprocess invocation — Claude Code now regenerates stale pages with full tool access (same capability that wrote the originals), no API key needed. Pipeline proven end-to-end: detect → regenerate → build → link-check → commit → push → deploy. Fast path completes in 8.7s.**

## What Happened

Two slices rewrote the regeneration engine from bare API calls to Claude Code subprocess invocation and proved it end-to-end.

**S01 (Claude Code Regeneration Engine)** fully rewrote `scripts/lib/regenerate-page.mjs`. The `@anthropic-ai/sdk` `client.messages.create()` call was replaced with `spawnSync('claude', ['-p', ...])` using flags for stream-json output, no session persistence, permission bypass, and sonnet model. The system prompt carries quality rules (section order, Mermaid styling, frontmatter format, link format) plus the exemplar page (capture.mdx). `findClaude()` detects CLI availability and returns false gracefully when absent. `parseStreamJson()` extracts model name and duration from subprocess output. `capDeps()` substitutes curated compiled JSON paths for reference pages with >50 dependencies. The `@anthropic-ai/sdk` package was removed from `package.json` and all import sites. A 20-test suite using a `mock-claude.sh` shell fixture (controlled by env vars) replaced the old SDK client-mock tests. Integration proof: `commands/capture.mdx` was regenerated by claude-sonnet-4-6 in 139.7s with valid frontmatter and all 6 required sections.

**S02 (Pipeline Integration and End-to-End Proof)** ran the full `npm run update` pipeline with 3 target stale pages. After restoring `reference/skills.mdx` (which had been accidentally deleted during S01 testing), the pipeline successfully regenerated all 3 pages: `commands/config.mdx` (291s), `reference/skills.mdx` (94s), `reference/extensions.mdx` (72s) via claude-sonnet-4-6. The build produced 65 pages, link check found 0 broken links in 4036. The worktree branch was merged to main (28 commits), pushed to origin, and the GitHub Actions deploy.yml workflow completed successfully (run 23235046096). A second `npm run update` with all pages fresh confirmed the fast path: regeneration completed in 2ms, pipeline-logic time was 8.7s.

## Forward Intelligence

### What the next milestone should know
- The regeneration pipeline is proven end-to-end. `npm run update` detects stale pages via SHA-based dependency hashing, spawns `claude -p` per stale page, builds, link-checks, stamps freshness, and the commit/push/deploy cycle works.
- Real regeneration takes 70–300s per page depending on complexity and API latency. Budget time accordingly.
- The `page-source-map.json` has 43 entries mapping pages to 778 source dependencies. All paths are structurally valid but not all mappings have been semantically verified (R051).
- 20 tests cover the regeneration engine via a `mock-claude.sh` shell fixture. The test pattern is: env vars control mock behavior (exit code, file writing, bad frontmatter, stdin capture).
- Token cost is no longer visible — subprocess mode reports model name and duration only. Duration is the only cost proxy.

### What's fragile
- `claude -p` occasionally deletes target files instead of rewriting them — observed specifically with `reference/skills.mdx` across multiple runs. No guard exists in the pipeline; the file is simply gone. A post-subprocess existence check with git restore fallback should be added.
- The 300s subprocess timeout is borderline for pages with many source dependencies under high API latency. `commands/config.mdx` took 291s in the successful run.
- `--dangerously-skip-permissions` flag is required for non-interactive subprocess invocation. If Claude Code changes this flag name or behavior, the subprocess will fail.
- `npm i -g gsd-pi@latest` adds ~12s of network latency to every `npm run update` run. Total wall-clock fast path is 20.7s vs 8.7s pipeline-logic time.

### Authoritative diagnostics
- `npm run update` output — the pipeline logs every step with timing and per-page regeneration shows ✓/✗/⊘ with model name and elapsed time. Single source of truth for pipeline health.
- `page-versions.json` — contains the dependency hash state for all 43 pages. Check this first for unexpected staleness.
- `node scripts/lib/regenerate-page.mjs <page>` — single-page debug with status, model, and duration.
- `node --test tests/regenerate-page.test.mjs` — 20-test suite validates all code paths.
- `gh run list --workflow=deploy.yml -L 1` — confirms latest GitHub Pages deployment status.

### What assumptions changed
- Original plan assumed per-page vs batch invocation strategy decision was needed — per-page invocation was the only strategy needed. No batch mode required.
- Original plan assumed page-source-map semantic verification would happen in S01 — it was descoped to focus on the engine rewrite and never picked back up in S02.
- `claude -p` file deletion behavior was unexpected — the subprocess occasionally deletes target files instead of rewriting them, requiring manual restoration.
- R055 fast path <15s was assumed to be total wall-clock — in practice, pipeline-logic time is 8.7s but `npm i -g gsd-pi@latest` adds ~12s. D054 redefines the metric.
