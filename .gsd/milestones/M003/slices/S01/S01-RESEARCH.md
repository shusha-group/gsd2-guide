# S01: Source Diff and Page Mapping — Research

**Date:** 2026-03-17

## Summary

S01 is a detection-only slice: build a system that knows which gsd-pi source files changed between versions, and which of the 42 authored doc pages are affected. The core insight from research is that **most of the change detection infrastructure already exists**. The GitHub manifest (`content/generated/manifest.json`) tracks 1025 repo files with SHA hashes, and `manifest.mjs` already has a `computeDiff()` function that returns `{ added, changed, removed }`. What's missing is: (1) a persistent source snapshot of the *previous* version's hashes for comparison, (2) the page-source-map that connects source file paths to doc page paths, and (3) a staleness resolver that combines the two.

The 42 authored pages fall into clear dependency categories. Command deep-dives (27 pages) each depend on a small, predictable set of source files — the command's `.ts` implementation file, its prompt `.md` file, and sometimes shared files like `commands.ts`, `state.ts`, or `paths.ts`. Recipe pages (6) and the walkthrough (1) are cross-cutting — they reference behaviors from multiple commands and don't map to single source files. Reference pages (6) depend on the extraction pipeline's JSON outputs. The changelog depends on GitHub releases data. The homepage depends on nothing dynamic.

## Recommendation

Build three deliverables:

1. **`content/generated/page-source-map.json`** — A hand-authored (initially) JSON manifest mapping each of the 42 doc page paths to an array of source file paths from the repo manifest. Generated by a new script `scripts/lib/build-page-map.mjs` that encodes the mapping rules. Cross-cutting pages (recipes, walkthrough) map to broad dependency sets — e.g. all prompt files, all template files, `commands.ts`, `state.ts`.

2. **`content/generated/source-snapshot/previous-manifest.json`** — A copy of the *previous* `manifest.json` saved before the extraction step overwrites it. The diff script compares this against the freshly-extracted manifest to detect changes.

3. **`scripts/lib/diff-sources.mjs`** — A module exporting `detectChanges(previousManifest, currentManifest)` and `resolveStalePages(changes, pageSourceMap)`. The former returns `{ changedFiles, addedFiles, removedFiles }`. The latter cross-references changed files against the page-source-map and returns `{ stalePages, reasons }`.

The page-source-map should be *generated from rules*, not purely hand-maintained. For the 27 command pages, the mapping is algorithmic: command name → `.ts` file + prompt `.md` file + shared deps. For the 15 non-command pages, the mapping is a static lookup table embedded in the script.

## Implementation Landscape

### Key Files

- `scripts/lib/manifest.mjs` — Already has `computeDiff(oldFiles, newFiles)` returning `{ added, changed, removed }`. This is the foundation. The new diff script wraps this, but also needs access to the *previous* manifest before it's overwritten.
- `scripts/extract.mjs` — The orchestrator. Currently overwrites `manifest.json` each run. Must be modified to save the previous manifest before building the new one (or the update pipeline must save it before calling extract).
- `scripts/update.mjs` — The pipeline orchestrator. Runs `npm update gsd-pi → extract → build → check-links`. The new diff + staleness step slots between extract and build (or is called within extract).
- `content/generated/manifest.json` — The current repo file manifest with 1025 files and SHA hashes. Paths use repo-relative format: `src/resources/extensions/gsd/auto.ts`, `src/resources/extensions/gsd/prompts/research-slice.md`, etc.
- `content/generated/commands.json` — 58 commands parsed from `docs/commands.md` markdown tables. Used by reference pages.
- `scripts/lib/extract-local.mjs` — Extracts skills/agents/extensions from the installed npm package. Its `resolvePackagePath()` function locates the gsd-pi installation.
- `scripts/lib/extract-commands.mjs` — Parses command tables from downloaded docs. Used to map command names to page slugs.

### Build Order

1. **`scripts/lib/build-page-map.mjs`** — Build this first. It encodes all 42 page-to-source mappings and writes `content/generated/page-source-map.json`. This is the core data structure. Verify by running it and inspecting the output — every authored `.mdx` page should appear, every source path should exist in the manifest.

2. **`scripts/lib/diff-sources.mjs`** — Build the diff + staleness resolver. It takes the previous manifest and current manifest, calls the existing `computeDiff()` logic, then cross-references against `page-source-map.json`. Verify by creating a mock scenario with known changed files and asserting the correct pages are flagged.

3. **Integrate into `scripts/extract.mjs` or `scripts/update.mjs`** — Save the previous manifest before extraction, then after extraction, run the diff + staleness resolver and report which pages are stale. This is the pipeline integration.

### Verification Approach

1. **Unit test for build-page-map:** Run the script, load the output JSON, verify:
   - All 42 authored page paths are present as keys
   - Every source path value exists in the current manifest (no typos)
   - Command pages have at least 1 `.ts` file and optionally prompt `.md` files
   - Cross-cutting pages (recipes, walkthrough) have > 3 source dependencies

2. **Unit test for diff-sources:** Create mock manifests with known differences:
   - Change `auto.ts` SHA → `auto.mdx` should be stale, plus any recipe/walkthrough that depends on auto
   - Add a new `.ts` file → should appear in `addedFiles` but not flag any existing pages as stale (S03 handles new commands)
   - Remove `doctor.ts` → should appear in `removedFiles` and flag `doctor.mdx` as stale

3. **Integration test:** Run the full pipeline with the current manifest as both "previous" and "current" → zero stale pages (no changes). Then manually tweak one SHA in the previous manifest → verify the correct pages are flagged.

4. **Manual review of 5 cross-cutting pages:** Verify the walkthrough and 4 recipe page mappings capture the essential source dependencies by reading the page content and confirming each referenced behavior traces to a mapped source file.

## Constraints

- The manifest uses repo-relative paths (`src/resources/extensions/gsd/auto.ts`), not installed-package paths (`dist/resources/extensions/gsd/auto.ts`). The page-source-map must use the same repo-relative path convention.
- The manifest is built from the GitHub tree API, not from local file hashing. This means change detection compares *repo* SHA hashes across `npm update` runs. If the repo changes but the npm package hasn't been published yet, no changes will be detected — this is correct behavior (we track what shipped, not what's in progress).
- The existing `manifest.json` is overwritten on every `npm run extract`. The previous version must be saved *before* extraction runs. The simplest approach: copy `manifest.json` → `previous-manifest.json` at the start of the extract step.
- `commands.json` is derived from `docs/commands.md` (a repo doc file), not from source code. Reference pages that depend on it should map to `docs/commands.md` in the manifest.
- `cli-flags.mdx` and `keyboard-shortcuts.mdx` reference runtime behavior defined in `commands.ts` (shortcut/flag registration) and `help-text.js` — but `help-text.js` lives at repo root as a compiled output, not under `src/resources/`. Map these pages to `commands.ts` as the best proxy.

## Common Pitfalls

- **Overwriting the previous manifest before saving it** — The extract pipeline must save the old manifest *before* `buildManifest()` runs. If this ordering is wrong, the diff will always show zero changes. Place the save at the top of `extract.mjs` or in `update.mjs` before the extract step.
- **Incomplete cross-cutting mappings** — Recipe and walkthrough pages reference behaviors from many commands. Missing a dependency means a source change won't flag the page as stale. Err on the side of including more dependencies — a false positive (regenerating a page unnecessarily) is cheap, a false negative (serving stale content) is a quality failure. Consider adding a "wildcard" dependency on `commands.ts` and `state.ts` for all cross-cutting pages.
- **Path format mismatch** — The manifest uses `src/resources/extensions/gsd/auto.ts` but the installed package has `dist/resources/extensions/gsd/auto.ts`. The page-source-map must consistently use the manifest's `src/` prefix format since that's what `computeDiff()` returns.

## Open Risks

- **Manifest staleness between extract runs** — The first time S01 runs after M002, there may be no previous manifest to compare against. The diff script must handle this gracefully: if no previous manifest exists, report "first run — all pages considered fresh" and skip staleness detection.
- **Repo restructuring** — If gsd-build/gsd-2 reorganizes its source tree (e.g., moves `src/resources/` to a different path), the page-source-map breaks. This is low probability but would require updating the mapping script. Consider logging a warning if > 50% of mapped source paths don't exist in the current manifest.
