## What It Does

Starting a new milestone on an existing GSD project is a single command: `/gsd new-milestone`. GSD reads your existing `.gsd/` context, detects what shipped previously, and walks you through scoping the next cycle — an interactive discussion where the LLM acts as a thinking partner, researches inline, writes `REQUIREMENTS.md`, and calls `gsd_plan_milestone` to produce the roadmap.

Prior milestone work stays completely intact. `PROJECT.md` carries the project history forward, completed milestone directories remain on disk, and the new milestone gets its own ID (M002, M003, etc.) with fresh slices inside it.

For multi-milestone visions, the discuss flow maps the full landscape and sets up each subsequent milestone — either discussing it now, writing a draft for later, or just queuing it for a future session. When the vision spans multiple milestones, a `DISCUSSION-MANIFEST.json` tracks each gate so auto-mode only starts when all milestones have been accounted for.

If you have a detailed spec ready in advance, drop it into `.gsd/runtime/headless-context.md` before running the command — GSD will create the milestone autonomously without any Q&A rounds.

## Usage

```
/gsd new-milestone
```

The command takes no arguments. The milestone label, scope, and structure are determined during the discuss flow.

### Step-by-Step

**1. Headless check** — GSD checks for `.gsd/runtime/headless-context.md`. If found, it reads the file, deletes it, and enters headless creation mode (no Q&A). Otherwise it shows the wizard and dispatches the discuss prompt when you select "Create next milestone."

**2. Reflect** — The LLM summarizes what it understood you want to build: a concrete description, honest size read (roughly how many milestones, how many slices in the first), a bullet list of major capabilities, and an invitation to correct. This step cannot be skipped.

**3. Investigate** — Before asking any questions, the LLM scouts the existing codebase (`rg`, `find`, `scout`), checks library docs for any referenced tech, and web searches unfamiliar domains. Questions that follow are grounded in what's actually true rather than assumptions.

**4. Question rounds** — The LLM asks 1–3 focused questions per round, following the depth checklist: what's being built, why, who it's for, what "done" looks like, the biggest technical unknowns, and what external systems it touches. Rounds continue until the checklist is fully satisfied — simple work may take 1–2 rounds, large visions may take 4+.

**5. Depth verification** — The LLM prints a structured summary of its understanding using your own terminology and framing, then asks you to confirm. This is the required write-gate — no files are written until depth is verified.

**6. Focused research** — Before writing the roadmap, the LLM researches table stakes the product space typically expects, domain-standard behaviors, likely omissions, and scope traps. Research findings are presented as candidate requirements — you confirm, defer, or reject. Nothing is silently added to scope.

**7. Multi-milestone mapping** — If the vision spans multiple milestones, the LLM proposes a milestone sequence (names, one-line intents, dependencies) and maps the full landscape before drilling into the first milestone. Subsequent milestones are handled per-milestone: discussed now, saved as `CONTEXT-DRAFT.md` for a future session, or just queued.

**8. Write REQUIREMENTS.md** — Requirements are written with stable `R###` IDs and organized into Active, Deferred, Out of Scope, and Traceability sections. Each requirement records its class, status, description, source, owning slice, and validation status. The full list is printed in chat before writing so you can catch omissions.

**9. Create roadmap** — The LLM calls `gsd_plan_milestone` with the decomposed slices, proof strategy, verification classes, requirement coverage, and boundary map. The tool persists the roadmap to the DB and writes `M001-ROADMAP.md`. The roadmap table is printed in chat before the tool is called.

**10. Write artifacts** — `M001-CONTEXT.md` is written preserving your exact terminology, emphasis, and framing from the discussion. Architectural or pattern decisions are saved via `gsd_decision_save`, which auto-assigns IDs and regenerates `.gsd/DECISIONS.md`.

**11. Multi-milestone gates (if applicable)** — For each remaining milestone, a per-milestone gate decides the readiness mode. `DISCUSSION-MANIFEST.json` is written after each gate. Auto-mode only starts when all gates are completed.

**12. Auto-mode starts** — The LLM says exactly "Milestone M001 ready." and auto-mode kicks off immediately.

### Headless Context Shortcut

Drop a spec into `.gsd/runtime/headless-context.md` before running `/gsd new-milestone`. GSD reads the file, deletes it, and creates the milestone without any Q&A rounds. The LLM still investigates the codebase and researches the domain — it just makes autonomous decisions rather than asking. Every judgment call is documented in `CONTEXT.md` under an "Assumptions" section.

This is the fastest path when you have detailed requirements already written and want planning to proceed unattended.

### Multi-Milestone Handling

When a vision spans multiple milestones, the first milestone gets full treatment (discuss, write CONTEXT.md, create roadmap). For each subsequent milestone, you choose its readiness mode:

| Mode | What happens | Downstream behavior |
|------|-------------|---------------------|
| **Discuss now** | Full discussion in the current session — writes `CONTEXT.md` | Auto-mode proceeds without pausing |
| **Draft for later** | Writes `CONTEXT-DRAFT.md` with seed material | Auto-mode pauses and prompts `/gsd` — wizard shows "Discuss from draft" |
| **Just queue it** | No context written — directory created | Auto-mode pauses and starts a fresh discussion |

All milestones that depend on others must have `depends_on: [M001, M002]` frontmatter in their `CONTEXT.md` — the state machine uses this to determine execution order.

### What Research Does

Research findings are advisory, not auto-binding. The LLM surfaces candidate requirements from domain analysis — things that are typically table stakes, common omissions, or known scope traps — and presents them explicitly for your decision. Nothing gets added to `REQUIREMENTS.md` without your confirmation.

In headless mode, the LLM makes autonomous inclusion/exclusion calls from the research and documents its reasoning.

## What Files It Touches

### Creates

| File | Purpose |
|------|---------|
| `.gsd/PROJECT.md` | Project description, current state, architecture, milestone sequence |
| `.gsd/REQUIREMENTS.md` | Capability contract with R### IDs, status, coverage, and traceability |
| `.gsd/DECISIONS.md` | Architectural decisions register — auto-generated by `gsd_decision_save` |
| `.gsd/DISCUSSION-MANIFEST.json` | Multi-milestone gate tracking — only written for multi-milestone discussions |
| `.gsd/milestones/M001/M001-CONTEXT.md` | Milestone intent, constraints, risks, and key decisions |
| `.gsd/milestones/M001/M001-ROADMAP.md` | Slices with risk, dependencies, demo lines, proof strategy, verification classes |
| `.gsd/milestones/M002/M002-CONTEXT.md` | Full context, draft, or none — depending on per-milestone gate decision |
| `.gsd/milestones/MXxx/MXxx-SECRETS.md` | Predicted external secrets with dashboard URLs and step-by-step instructions (if needed) |

### Reads

| File | Purpose |
|------|---------|
| `.gsd/PROJECT.md` | Existing project state and milestone history (if any) |
| `.gsd/REQUIREMENTS.md` | Existing requirements — continues numbering from here |
| `.gsd/runtime/headless-context.md` | Pre-gathered spec — optional, triggers headless creation if present |
| Codebase files | Investigated during the mandatory codebase scouting step |

### Deletes

| File | Purpose |
|------|---------|
| `.gsd/runtime/headless-context.md` | Consumed immediately after reading — not preserved |

## Examples

**M001 complete, starting M002:**

```
> /gsd new-milestone

● GSD — Get Shit Done
  No active milestone.
  ┌─────────────────────────────────────────────────────────┐
  │ ● Create next milestone — Define what to build next.    │
  └─────────────────────────────────────────────────────────┘
```

After selecting "Create next milestone", the discuss flow begins:

```
What's the vision?

> I want to add push notifications. Users should get alerts when
> someone comments on their post or follows them. Also want an
> in-app notification center so they can review past alerts.
```

The LLM reflects first:

```
Here's what I'm hearing:

- Push alerts for social events (comments, follows)
- In-app notification center with history
- Per-type opt-out controls (implied by "alerts")

Size read: 1 milestone, 3–4 slices. Manageable.

Here's my read. Correct anything important I missed.
```

After investigation, question rounds, and depth verification, the LLM researches the domain:

```
Focused research complete.

Table stakes for a notification system:
- Delivery receipts (did it arrive?)
- Badge counts on app icon
- Per-type opt-out per user

Candidate requirements to add:
- R004: User can opt out of individual notification types
- R005: Notification badge count reflects unread count

Include / defer / out of scope?
```

**Headless creation from a spec file:**

```bash
# Write your spec
cat > .gsd/runtime/headless-context.md << 'EOF'
# M002: Notification System

Add push notifications for social events (comments, follows) and an
in-app notification center. Users must be able to opt out per type.

Stack: React Native + Expo, Supabase backend.
Target: iOS and Android.
EOF

# Run — no Q&A, creates milestone autonomously
/gsd new-milestone
```

GSD reads the spec, scouts the codebase, researches the domain, documents assumptions, and creates `M002-CONTEXT.md` and `M002-ROADMAP.md` without any interaction. Auto-mode starts when done.
